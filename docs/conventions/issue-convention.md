# Issue Convention

## 목적

이슈는 논의 저장소가 아니라 실행 가능한 작업 단위를 정의하는 문서여야 합니다.
이 저장소에서는 기능 제안, 버그 제보, 작업 계획을 구분해서 기록합니다.

## 템플릿 선택 기준

- `버그 제보`: 현재 구현이 기대와 다르게 동작하거나 회귀가 발생했을 때
- `기능/구조 제안`: 새로운 기능, 설계 변경, 구조 개선이 필요할 때
- `작업 계획`: 문서, 빌드, 설정, 운영 작업처럼 구현 전후의 독립 작업을 정리할 때

## 제목 규칙

이슈 제목은 템플릿 prefix를 유지합니다.
prefix는 영어 소문자로 작성합니다.

- `[bug]` 실제 문제를 한 줄로 설명
- `[proposal]` 원하는 변화 또는 설계 방향을 한 줄로 설명
- `[task]` 완료되었을 때의 상태를 한 줄로 설명

예시:

- `[bug] QEMU boot stops before VGA output`
- `[proposal] split serial and VGA console responsibilities`
- `[task] document make help and validation flow`

## 작성 규칙

- 한 이슈는 하나의 목표만 가집니다.
- 배경에는 현재 문제를, 제안에는 원하는 결과를 씁니다.
- 범위에는 포함과 제외를 모두 적습니다.
- 완료 조건은 체크리스트로 작성합니다.
- 구현자가 바로 시작할 수 있을 만큼 정보를 남깁니다.

## 버그 이슈 필수 정보

- 실제 증상
- 재현 절차
- 기대 동작
- 확인 환경
- 로그, 화면 출력, 커밋 해시 같은 증거

커널/부트 관련 버그는 가능하면 아래 중 하나 이상을 포함합니다.

- `make build` 결과
- `make iso` 결과
- QEMU 직렬 출력
- 화면 캡처 또는 부팅 단계 설명

## 제안 이슈 필수 정보

- 왜 필요한지
- 이번 이슈에 포함할 범위와 제외할 범위
- 완료 조건
- 예상 작업 목록

아이디어만 있고 완료 조건이 없다면 아직 이슈 크기가 정리되지 않은 상태로 봅니다.

## 작업 계획 이슈 규칙

- 구현 작업을 시작하기 전 준비 단계 정리에 사용합니다.
- 한 번의 PR로 끝낼 수 있는 크기로 나눕니다.
- 후속 작업이 필요하면 새 이슈로 분리합니다.

## 좋은 이슈의 기준

- 구현자와 리뷰어가 같은 완료 조건을 공유할 수 있다.
- 범위 밖 작업이 무엇인지 명확하다.
- PR 설명을 다시 쓰지 않아도 될 만큼 배경이 정리돼 있다.

## 피해야 할 패턴

- 여러 문제를 한 이슈에 모으기
- "정리하기", "개선하기"처럼 완료 상태가 모호한 제목
- 재현 방법 없는 버그 제보
- 제외 범위 없는 구조 제안
